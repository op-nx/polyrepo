    ## Background

    The UAT reported that `preTasksExecution` never fires — POLYREPO_HASH_<ALIAS> env vars are never set. The UAT attributed this to "PreTasksExecution API does not exist in Nx 22.5.4." However, independent analysis of the installed Nx 22.5.4 source confirms the full runtime infrastructure IS present:

    - `public-api.d.ts` exports `PreTasksExecution` type (line 115)
    - `NxPluginV2` interface includes `preTasksExecution?` (line 94)
    - `loaded-nx-plugin.js` wraps plugin's `preTasksExecution` with Proxy env capture (lines 61-80)
    - `load-resolved-plugin.js` checks `'preTasksExecution' in m.default` (line 17)
    - `tasks-execution-hooks.js` exports `runPreTasksExecution` which loads plugins and calls hooks
    - `run-command.js` calls `runPreTasksExecution` at line 290, before `runCommandForTasks` at line 297

    The root cause may be one of:
    a) The TS transpiler that Nx uses for local plugins doesn't propagate the `preTasksExecution` export correctly
    b) The module loader's `m.default` check in `load-resolved-plugin.js` (line 12-20) doesn't find the export — CJS named exports live on `m` directly, not `m.default`. The fallback `return m` (line 21) should handle this, but there may be a subtlety.
    c) The daemon's plugin cache holds stale LoadedNxPlugin instances that don't have `preTasksExecution` (from before Plan 02 was implemented).
    d) `getPlugins` module-level caching in `tasks-execution-hooks.js` returns plugins loaded during graph computation that somehow lost the `preTasksExecution` handler.

    ## Fix Implementation

    Based on diagnostic findings, apply ONE of these fixes:

    **Fix A (most likely needed): Explicit default export**
    If `load-resolved-plugin.js` only detects the plugin via `m.default` and the TS compilation doesn't produce a default export, add:
    ```typescript

    const plugin = { createNodesV2, createDependencies, preTasksExecution };
    export default plugin;
    ```
    Note: Nx's `importPluginModule` checks `m.default` first (lines 12-18), then falls back to `m` (line 21). If both paths work, the default export is preferred. This fix ensures the preTasksExecution export is discoverable via either path.

    **Fix B: Alternative env var injection via createNodesV2**
    If `preTasksExecution` genuinely cannot be made to work (e.g., it only fires during `nx run` but not during `nx run-many`, or the daemon IPC boundary silently drops the env vars), move the git hash computation into `createNodesV2`. The env vars would be set during graph computation. This is suboptimal (graph computation is cached separately), but it would work because the graph computation runs on every CLI invocation when the daemon is disabled, and the daemon re-runs it when the project graph changes.

    IMPORTANT: Fix B is a LAST RESORT. Try Fix A first. The user's CONTEXT.md locked decision specifies `preTasksExecution` as the mechanism.

    **Fix C: Runtime inputs as fallback (with NX_DAEMON=false disclaimer)**

This works correctly with `NX_DAEMON=false` but is broken by nrwl/nx#30170 with daemon enabled. Document this limitation and recommend users set `NX_DAEMON=false` until the upstream bug is fixed.

    IMPORTANT: Fix C breaks the user's locked decision on PROXY-05 (daemon-safe caching). Only use if A and B both fail.
